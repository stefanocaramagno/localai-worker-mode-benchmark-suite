# 09. Metrics, Artifacts, and Cluster-Side Evidence

## Objective

This runbook defines the mandatory evidence model for every benchmark and validation run in the pipeline. It standardizes what must be collected, where artifacts must be stored, how client-side and cluster-side evidence are linked to a run, and what must be verified before a run can be considered traceable and technically defensible.

In the current repository version, the ordinary benchmark launchers already invoke the metric-set and cluster-side collection steps automatically as part of the run protocol. Therefore, this runbook should be read primarily as the authoritative evidence contract for the pipeline, not as an instruction to manually rerun the same collection commands during every normal execution.

This document applies to:

- official baseline execution;
- exploratory validation runs;
- pilot sweep executions;
- consolidated pilot benchmark campaigns;
- any rerun intended to replace or validate a previous result set.

This runbook does **not** define the benchmark logic itself. It defines the evidence layer that must accompany every execution.

---

## Why This Runbook Matters

Client-side latency and throughput values are not sufficient on their own.

For each run, the pipeline must preserve enough evidence to answer all of the following questions after the fact:

- Which scenario was executed?
- Which model, worker count, placement policy, and workload configuration were active?
- Which exact run identifier produced the results?
- What happened on the cluster before the measurement phase?
- What happened on the cluster after the measurement phase?
- Where were the pods scheduled?
- Were resource metrics available and captured?
- Were there restarts, placement anomalies, or event-level warnings that help explain the result?

Without this evidence, benchmark results may still exist, but they cannot be reconstructed, audited, or interpreted with sufficient rigor.

---

## Scope

This runbook covers:

- the minimum mandatory metric set;
- the artifact naming rules tied to the run convention;
- client-side result files generated by Locust-based runs;
- cluster-side evidence captured with `kubectl`;
- metric-set manifest generation;
- cluster-side capture manifests and summaries;
- recommended validation checks on saved artifacts.

This runbook does **not** replace:

- precheck and smoke validation;
- baseline execution procedures;
- sweep execution procedures;
- statistical validation and diagnosis procedures.

Those topics are covered by separate runbooks.

## Prerequisites

Before relying on a run as evidence-complete, verify the following baseline conditions for the collection path used by the active launcher or by any explicitly requested manual recovery action:

1. The run has a valid output root or CSV prefix.
2. The run identifier has been generated according to `RC1`.
3. The target namespace is known.
4. `kubectl` is available in the execution environment.
5. The cluster-side collection script can access the cluster through the intended kubeconfig.
6. The benchmark launcher has completed or is about to complete a measurement phase.
7. The measurement output directory is writable.

If any of these conditions is not met, do not proceed with evidence collection.

## Mandatory Evidence Model

Every valid run must preserve **both** of the following evidence classes:

### 1. Client-Side Evidence

This evidence describes what the load generator observed.

Mandatory categories:

- request count;
- success rate;
- failure count;
- mean response time;
- p50 response time;
- p95 response time;
- p99 response time;
- throughput.

### 2. Cluster-Side Evidence

This evidence describes what the platform looked like around the run.

Mandatory categories:

- node CPU usage;
- node memory usage;
- pod CPU usage;
- pod memory usage;
- pod placement;
- pod restart count;
- pod readiness;
- service inventory;
- namespace events;
- pod descriptions.

A run is incomplete if it includes only one of these two categories.

## Run Convention and Artifact Naming

The canonical run convention is defined in `config/conventions/RC1.json`.

### Key Rules

- the run identifier must be unique;
- the run identifier must encode campaign, primary scenario, and replica;
- all output files must be directly traceable to the run identifier;
- all measurement outputs must be generated from a shared CSV prefix.

### Run ID Format

The repository defines the following logical format:

`RID-<UTC_TIMESTAMP_COMPACT>-<CAMPAIGN>-<PRIMARY_SCENARIO>-R<REPLICA>`

### CSV Prefix Rule

All execution flows must use a CSV prefix derived from the run identifier.

This rule ensures that the following artifact families remain aligned:

- Locust client-side files;
- protocol manifests;
- phase manifests;
- metric-set manifests;
- cluster-side pre-capture evidence;
- cluster-side post-capture evidence.

## Client-Side Artifacts

The metric-set library defines the canonical Locust client-side artifacts as:

- `<measurement_csv_prefix>_stats.csv`
- `<measurement_csv_prefix>_stats_history.csv`
- `<measurement_csv_prefix>_failures.csv`
- `<measurement_csv_prefix>_exceptions.csv`

These files are the primary client-side source artifacts referenced by the metric-set manifest.

### Expected Meaning

#### `_stats.csv`
Primary summary table for request-level performance and aggregate counters.

#### `_stats_history.csv`
Time-series view of request behavior over the benchmark window.

#### `_failures.csv`
Failure-level breakdown, if failures occurred.

#### `_exceptions.csv`
Runtime exception output produced during load execution, if any.

### Client-Side Validation Rules

At minimum, verify that:

- `_stats.csv` exists for every completed measurement run;
- `_stats_history.csv` exists when the run was launched with `--csv-full-history`;
- `_failures.csv` and `_exceptions.csv` are present even if empty, or their absence is justified by the specific launcher behavior;
- the files share the same measurement prefix.

If the client-side files do not share a consistent prefix, treat the run as suspicious and review the launcher invocation.

## Cluster-Side Artifacts

The canonical cluster-side collection profile is defined in `config/cluster-capture/CS1.json`.

It requires the following artifact families for **both** capture stages: `pre` and `post`.

### Cluster-Side Artifact Set

For each stage prefix, the collector generates:

- `<stage_prefix>_manifest.json`
- `<stage_prefix>_summary.txt`
- `<stage_prefix>_nodes-wide.txt`
- `<stage_prefix>_top-nodes.txt`
- `<stage_prefix>_pods-wide.txt`
- `<stage_prefix>_top-pods.txt`
- `<stage_prefix>_services.txt`
- `<stage_prefix>_events.txt`
- `<stage_prefix>_pods-describe.txt`

### Meaning of Each File

#### `_manifest.json`
Machine-readable description of the cluster-side capture:

- profile ID;
- namespace;
- stage;
- artifact list;
- exact `kubectl` commands used.

#### `_summary.txt`
Human-readable summary of the cluster-side capture.

#### `_nodes-wide.txt`
Output of:

`kubectl get nodes -o wide`

Used to preserve node inventory and node-level placement context.

#### `_top-nodes.txt`
Output of:

`kubectl top nodes`

Used to preserve node CPU and memory usage at capture time.

#### `_pods-wide.txt`
Output of:

`kubectl get pods -n <namespace> -o wide`

Used to preserve pod placement, pod state, and node assignment.

#### `_top-pods.txt`
Output of:

`kubectl top pods -n <namespace>`

Used to preserve pod-level CPU and memory usage.

#### `_services.txt`
Output of:

`kubectl get svc -n <namespace>`

Used to preserve the namespace service inventory during the run.

#### `_events.txt`
Output of:

`kubectl get events -n <namespace> --sort-by=.metadata.creationTimestamp`

Used to preserve warnings, restarts, scheduling decisions, and other event-level signals.

#### `_pods-describe.txt`
Output of:

`kubectl describe pods -n <namespace>`

Used to preserve pod conditions, restarts, mounts, images, requests, limits, probes, and scheduling reasons.

## Pre-Stage and Post-Stage Capture

Cluster-side evidence is collected in two separate stages.

### Pre-Stage Capture

Purpose:

- capture the cluster state immediately before the benchmark window;
- confirm pod placement and node state before measurement;
- preserve early signals such as pending pods, readiness issues, or resource pressure.

### Post-Stage Capture

Purpose:

- capture the cluster state immediately after the benchmark window;
- preserve post-run CPU and memory conditions;
- capture restarts, scheduling changes, or warning events triggered during the run.

### Why Both Stages Are Required

A single cluster snapshot is not sufficient.

You need both:

- a pre-run state to establish execution conditions;
- a post-run state to explain outcomes.

If only post-run evidence exists, the initial conditions are missing.
If only pre-run evidence exists, the benchmark impact is missing.

## How Cluster-Side Collection Is Invoked

In the current pipeline, cluster-side collection is ordinarily triggered by the official launchers and protocol helpers. Under normal benchmark execution, operators should therefore expect the required pre-stage and post-stage artifacts to be produced automatically by the active flow.

The direct script invocations below are kept only as:

- reference examples of the underlying collector interface;
- fallback commands for targeted recovery when a launcher step must be rerun in isolation;
- troubleshooting tools for controlled operator intervention.

They are **not** the default operational procedure for every standard run.

### PowerShell Reference / Recovery Invocation

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -File ".\scripts\validation\cluster-side\Collect-ClusterSideArtifacts.ps1" `
  -ProfileConfig .\config\cluster-capture\CS1.json `
  -Kubeconfig .\config\cluster-access\kubeconfig `
  -Namespace genai-thesis `
  -OutputPrefix .\results\some-run\RID-..._cluster_pre `
  -Stage pre
```

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -File ".\scripts\validation\cluster-side\Collect-ClusterSideArtifacts.ps1" `
  -ProfileConfig .\config\cluster-capture\CS1.json `
  -Kubeconfig .\config\cluster-access\kubeconfig `
  -Namespace genai-thesis `
```

If the active benchmark launcher has already produced the canonical cluster-side artifacts for the run, do not rerun the collector manually just to duplicate the same evidence set. Manual invocation is justified only when:

- the launcher-based capture failed or is incomplete;
- recovery is needed while the relevant benchmark state is still valid and interpretable;
- a controlled troubleshooting step has been explicitly requested.

## Metric-Set Manifest Generation

The metric-set library is responsible for linking the canonical metric vocabulary to the physical artifacts produced by a run.

The metric-set manifest and text summary are generated from the `MS1` profile.

### Expected Outputs

- `<measurement_csv_prefix>_metric-set.json`
- `<measurement_csv_prefix>_metric-set.txt`

### What the Metric-Set Manifest Must Contain

- metric-set profile ID;
- launcher name;
- run ID;
- measurement CSV prefix;
- minimum client-side metric list;
- minimum cluster-side metric list;
- client-side source artifact list;
- cluster-side source artifact list.

This file is essential because it turns the abstract metric contract into a concrete set of files attached to a specific run.

## Relationship Between Protocol, Metric Set, and Cluster Capture

The repository already models these three concerns separately.

### `EP1` Protocol Profile
Defines the ordered benchmark execution steps.

### `MS1` Metric Set Profile
Defines the mandatory metric vocabulary and the expected source artifact classes.

### `CS1` Cluster Capture Profile
Defines the required cluster-side files and stage naming.

### Correct Interpretation

- `EP1` answers: **what phases must run and in what order**;
- `MS1` answers: **what must always be measurable and preserved**;
- `CS1` answers: **how platform evidence must be captured**.

This separation is deliberate and must remain intact.

## Validation Checklist for Every Run

After a run completes, validate the following before accepting it.

### Identity and Naming

- The run ID exists and matches the campaign/scenario/replica.
- All major artifacts share the same measurement prefix.
- The cluster pre/post stage prefixes are aligned with the same run.

### Client-Side Evidence

- `_stats.csv` exists.
- `_stats_history.csv` exists.
- Failure and exception outputs are preserved.
- The files are not empty unless a documented reason exists.

### Cluster-Side Evidence

- Pre-stage cluster capture exists.
- Post-stage cluster capture exists.
- Both stages contain a manifest and summary.
- `pods-wide` and `top-pods` exist for both stages.
- `events` and `pods-describe` exist for both stages.

### Metric-Set Evidence

- `_metric-set.json` exists.
- `_metric-set.txt` exists.
- The metric-set manifest references both client-side and cluster-side artifact lists.

### Interpretability Evidence

- Pod placement can be reconstructed from the saved files.
- Resource pressure can be reviewed from `top-nodes` and `top-pods`.
- Event-level explanations can be checked in `_events.txt`.
- Restart and readiness details can be checked in `_pods-describe.txt`.

If any of these checks fails, the run must be flagged for review.

## Common Failure Modes

### 1. Client-Side Results Exist but Cluster-Side Evidence Is Missing

Impact:

- latency and throughput are available;
- infrastructure-level explanation is missing.

Action:

- do not treat the run as fully valid for comparative analysis;
- rerun the scenario if the missing evidence affects interpretation.

### 2. Pre-Stage Capture Exists but Post-Stage Capture Is Missing

Impact:

- initial state is known;
- benchmark consequences on the cluster are not preserved.

Action:

- treat the run as incomplete;
- do not use it as a reference run for diagnosis.

### 3. Post-Stage Capture Exists but Pre-Stage Capture Is Missing

Impact:

- end-state is known;
- initial execution conditions are unknown.

Action:

- treat the run as incomplete.

### 4. `kubectl top` Fails During Collection

Impact:

- pod and node usage metrics are incomplete or unavailable.

Typical causes:

- metrics pipeline unavailable;
- `metrics-server` not functioning correctly;
- temporary cluster-side issue.

Action:

- investigate cluster metrics availability;
- rerun evidence collection only if it preserves the intended benchmark state;
- otherwise rerun the benchmark under healthy conditions.

### 5. Artifact Prefix Mismatch

Impact:

- traceability is broken;
- files from different runs may be mixed.

Action:

- reject the run as improperly recorded;
- fix the launcher or prefix generation logic before continuing.

## Operational Recommendations

### Always Keep Evidence Generation Enabled

Do not disable metric-set generation or cluster-side capture in normal benchmark campaigns. The ordinary expectation is that the official launchers keep these steps enabled and execute them automatically as part of the standard protocol.

### Preserve Raw Artifacts

Do not overwrite or manually edit generated raw artifacts after a run.

### Prefer Reviewable Failure Over Silent Omission

If a cluster-side collector step fails, surface the error and stop the run classification process instead of silently accepting partial evidence. If recovery is attempted manually, record that intervention explicitly in the operator notes for the affected run.

### Keep Pre and Post Captures Close to the Measurement Window

Do not insert long uncontrolled delays between:

- benchmark completion;
- post-stage cluster-side collection.

The value of post-stage evidence decreases as time drifts away from the run.

### Use the Same Namespace That Was Benchmarked

Never collect cluster-side evidence from a different namespace than the one used by the scenario.

---

## Acceptance Criteria

A run can be considered evidence-complete only if all the following are true:

1. A valid run ID exists.
2. Client-side Locust artifacts are present.
3. A metric-set manifest exists.
4. Pre-stage cluster-side artifacts are present.
5. Post-stage cluster-side artifacts are present.
6. Artifact naming is consistent across the run.
7. Pod placement can be reconstructed from saved evidence.
8. Resource usage can be inspected at both node and pod level.
9. Events and pod descriptions are preserved for troubleshooting and diagnosis.

If any of these conditions is false, the run may still be useful for debugging, but it must not be treated as a fully valid benchmark record.
