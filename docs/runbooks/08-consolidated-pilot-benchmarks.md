# 08. Consolidated Pilot Benchmarks

## Objective

This runbook defines the controlled execution process for the consolidated pilot benchmark campaign. Its purpose is to re-run the pilot sweeps under a stricter and repeatable protocol so that preliminary benchmark evidence can be converted into a more reliable baseline for technical diagnosis and phase closure.

The consolidated campaign covers the following scenario families:

- `worker-count`
- `workload`
- `models`
- `placement`

Each family is executed under the same controlled execution stack, with fixed warm-up and measurement windows, mandatory A/B/C replicas, cluster-side evidence collection, and statistical rigor summarization.

---

## When to Execute This Runbook

Execute this runbook only after all of the following conditions are met:

- the cluster access and bootstrap steps are complete;
- the repository execution environment is ready;
- the official baseline has been defined and validated;
- the exploratory and pilot sweeps have already been exercised at least once;
- the precheck and API smoke validation flow is stable;
- the LocalAI worker-mode topology is deployable without unresolved blocking issues.

This runbook does **not** replace exploratory validation. It is the formal benchmark re-execution phase used to produce comparable, more stable evidence.

---

## Scope and Expected Outcome

The consolidated pilot campaign is intended to produce:

- a campaign manifest describing the complete execution set;
- a campaign text report with the executed commands and campaign status;
- one execution per scenario replica (`A`, `B`, `C`) for every configured family;
- per-run benchmark artifacts generated by the underlying pilot launcher;
- a statistical rigor summary over the primary measurement CSV files;
- a campaign status that can be used by the post-analysis stages.

The campaign is considered operationally valid only if the selected family scope is executed under the controlled protocol and the expected campaign artifacts are generated. In the current implementation, the final campaign status can be either `success` or `completed_with_unsupported_scenarios`, depending on whether one or more scenarios are formally classified as `unsupported_under_current_constraints`.

## Governing Configuration

### Consolidated Pilot Profile

The campaign is governed by:

- `config/pilot-consolidation/CP1.json`

This profile defines:

- the consolidated output root;
- the mandatory precheck, phase, protocol, cluster-capture, metric-set, and statistical-rigor profiles;
- the controlled warm-up and measurement durations;
- the scenario families to execute;
- the required scenario lists per family;
- the required replica set per family;
- stop-on-first-failure behavior.

The current profile sets:

- `outputRoot`: `results/pilot/consolidated`
- `warmUpDuration`: `45s`
- `measurementDuration`: `3m`
- `stopOnFirstFailure`: `true`

### Statistical Rigor Profile

The campaign also uses:

- `config/statistical-rigor/SR1.json`

This profile defines:

- the required replica count;
- the minimum successful replica count;
- the cooldown between replicas;
- the cooldown between scenarios;
- the stabilization timeout between completed runs;
- the stabilization polling interval used while checking cluster convergence;
- the tracked metrics used in the rigor summary;
- the variability thresholds used to evaluate consistency.

The current profile requires:

- `requiredReplicaCount = 3`
- `minimumSuccessfulReplicaCount = 3`
- `coolDownBetweenReplicasSeconds = 10`
- `coolDownBetweenScenariosSeconds = 20`
- `stabilizationTimeoutSeconds = 180`
- `stabilizationPollIntervalSeconds = 5`

Tracked metrics include:

- request count;
- failure count;
- mean response time;
- p50 response time;
- p95 response time;
- p99 response time;
- throughput.

## Prerequisites

Before starting the campaign, verify all of the following.

### Cluster and Access

- VPN connection is active.
- The expected `kubeconfig` file is available and valid.
- The target Kubernetes context is correct.
- All required nodes are in `Ready` state.
- The target namespace is reachable.

### Deployment Readiness

- The LocalAI worker-mode deployment flow is already validated.
- API smoke validation succeeds.
- The selected baseline model is available.
- The cluster has enough free resources for the selected consolidated family.

### Repository Readiness

- The repository root is the current working directory, or all command paths are adjusted accordingly.
- Required local tools are installed and available in `PATH`.
- The `results/` tree is writable.
- No stale artifacts from a previous failed campaign are going to be confused with new outputs.

### Methodological Readiness

- The baseline and scenario governance have already been frozen.
- No uncontrolled scenario edits are pending.
- The family being consolidated is ready for re-execution.
- Any known scenario limits are already understood and accepted as part of the benchmark design.

## Execution Model

For each family, the consolidated launcher performs the following high-level sequence:

1. resolve the campaign profile;
2. resolve the statistical rigor profile;
3. create a campaign identifier and output paths;
4. iterate through each scenario configured for the family;
5. execute replicas `A`, `B`, and `C` for that scenario;
6. if a replica completes normally, wait for cluster stabilization before continuing;
7. if a replica is classified as `unsupported_under_current_constraints`, record the unsupported outcome and skip success-like stabilization for that entry;
8. apply replica cooldown between consecutive replicas;
9. apply scenario cooldown between consecutive scenarios;
10. collect campaign entries and locate the corresponding measurement CSV files;
11. generate a campaign manifest and campaign text report;
12. generate a statistical rigor manifest and text summary.

By default, the profile is configured to stop on the first failure.

## Required Success Conditions

The consolidated pilot campaign should be treated as operationally valid only if all of the following are true:

- every required scenario in the selected family scope was processed by the campaign;
- replicas `A`, `B`, and `C` exist for every scenario unless execution stops earlier because of the configured failure policy;
- the corresponding measurement CSV files are available for replicas that completed normally;
- unsupported scenarios are explicitly recorded as `unsupported_under_current_constraints` and are not silently mixed with successful replicas;
- the campaign manifest and rigor summary were generated successfully;
- the campaign status is either `success` or `completed_with_unsupported_scenarios`;
- no unclassified infrastructure anomaly invalidated the campaign.

If a required scenario fails unexpectedly and stop-on-first-failure is active, the campaign must be considered incomplete until re-run. If a scenario is formally classified as unsupported under the frozen baseline constraints, that outcome is part of the campaign evidence and must be recorded as such rather than re-labeled as a generic execution failure.

## Operational Recommendations

### Run One Family at a Time During Controlled Operations

Although the launcher supports `all`, the most defensible operational mode is to consolidate one family at a time. This improves observability, reduces troubleshooting ambiguity, and makes reruns easier.

### Preserve the Frozen Baseline

Do not edit baseline assumptions while a consolidated family is in progress. The purpose of this phase is to produce consistent evidence, not to iterate on scenario design.

### Keep the Cluster Quiet Between Replicas

Respect the configured cooldown windows. They are part of the controlled execution regime and should not be bypassed unless the profile is intentionally updated.

### Treat Failures and Unsupported Outcomes as Signal

Do not silently discard failed replicas. If a scenario fails repeatedly under the controlled consolidated regime, that result may be part of the technical diagnosis.

Likewise, if a scenario is classified as `unsupported_under_current_constraints`, keep that result in the campaign record. Under the current implementation, unsupported scenarios are valid evidence about the practical limits of the frozen baseline and should not be rewritten as successful runs.

## Common Failure Modes

### 1. Precheck Failure

Symptoms:

- node not ready;
- namespace not reachable;
- metrics not available;
- service endpoint not available.

Action:

- stop the campaign;
- resolve the precheck issue first;
- restart the family from a clean state.

### 2. API Smoke Failure

Symptoms:

- LocalAI endpoint does not respond correctly;
- model lookup fails;
- payload validation fails.

Action:

- inspect the deployment and service exposure;
- verify model availability;
- re-run smoke validation before restarting the family.

### 3. Missing Measurement CSV

Symptoms:

- rigor summary cannot resolve the expected stats CSV;
- per-run benchmark appears to complete but no measurement file is found.

Action:

- inspect the family output tree;
- verify that the underlying pilot launcher produced the expected Locust artifacts;
- do not accept the scenario as successful without the measurement CSV.

### 4. Stop-on-First-Failure Triggered

Symptoms:

- campaign stops mid-family;
- manifest contains a failed entry;
- later scenarios are not executed.

Action:

- inspect the first failing replica;
- determine whether the failure is infrastructural, scenario-specific, or methodological;
- re-run the family after correction.

### 5. Unsupported Scenario Recorded

Symptoms:

- a scenario entry is marked as `unsupported_under_current_constraints`;
- delegated launcher diagnostics indicate unschedulable pods, insufficient resources, or placement constraints;
- the consolidated campaign records the unsupported result and skips success-like stabilization for that entry.

Action:

- inspect the unsupported report and the corresponding cluster-side artifacts;
- confirm that the unsupported outcome is consistent with the frozen baseline constraints;
- treat the result as campaign evidence, not as a generic launcher crash.

### 5. Excessive Variability Between Replicas

Symptoms:

- rigor summary shows high coefficient of variation on mean response time, p95, or throughput.

Action:

- inspect the cluster-side evidence and timing of the three replicas;
- verify that no uncontrolled environment drift occurred;
- re-run the scenario if needed under a cleaner cluster state.

## Post-Execution Verification Checklist

After each consolidated family, confirm the following:

- campaign files exist under `results/pilot/consolidated/campaigns`;
- unsupported scenarios, if any, are explicitly visible in the campaign manifest and status;
- scenario output trees exist under the selected family directory;
- three replicas are present per required scenario;
- the rigor summary file exists and is readable;
- no unexpected partial run remains unclassified;
- any failure is already linked to a concrete follow-up action.

## Exit Criteria for This Phase

- the family has been executed under the consolidated profile;
- the required A/B/C replicas have been generated;
- the statistical rigor summary has been created;
- the family artifacts are ready for technical diagnosis.

The broader consolidated phase is complete when the same outcome has been achieved for all four pilot families.
